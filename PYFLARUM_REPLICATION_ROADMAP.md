这份文档用于统一 `pyflarum` 后续开发的评估口径、差距清单、迭代顺序和开发原则，避免继续以“页面已经有了”或“字段已经存了”来误判完成度。

- 产品体验对齐优先于技术实现对齐。

- “有页面但不能用”、“有设置但不生效”、“有接口但没有前端入口”，都只能算部分完成。

## 3. 总体结论

### 3.1 当前项目定位

`pyflarum` 当前已经不是“论坛骨架”，而是一个具备较强可用性的 Flarum 风格论坛系统；但它距离“Flarum 2.x 官方产品体验的 Python 复刻版”仍有明确差距。

它更接近：

- 一个已有较完整核心论坛链路的 Django + Vue 社区产品
- 一个部分复刻了 Flarum 前台体验与后台结构的项目

它还不接近：

- README 所写的“100% 完成”或“完美复刻”
- Flarum 2.x 官方内置产品体验的完整等价实现

README 当前口径见：[README.md](/D:/files/project/tmp/pyflarum/README.md#L1)

### 3.2 完成度估算

以下是按“Flarum 2.x 官方产品体验”口径给出的保守估算：

| 维度 | 当前估算 | 判断 |
| --- | --- | --- |
| 前台核心论坛体验 | 75% - 80% | 讨论、帖子、基础通知、搜索、Composer 已较完整，但细节与治理能力仍弱 |
| 后台产品化能力 | 55% - 65% | 后台页面齐全度不错，但不少模块仍停留在基础 CRUD 或设置持久化 |
| 官方扩展能力覆盖 | 45% - 55% | `likes`/`lock`/`sticky`/`subscriptions` 较接近，多个官方扩展仍明显缺失 |
| 工程成熟度 | 60% - 70% | 后端测试已有基础，但缺少前端测试、E2E、契约测试和更强运行时验证 |
| 整体对标 Flarum 2.x | 65% - 70% | 已有明显成果，但还不能宣称“完整复刻” |

结论：

- 可以把当前项目视为“已完成约三分之二的 Flarum 2.x Python 化产品复刻”
- 下一阶段重点不应再是泛泛补页面，而应是“补闭环、补扩展、补治理、补运行时、补测试”

## 5. PyFlarum 与 Flarum 2.x 的主要差距

以下差距是当前最需要正视的部分。

### 5.1 核心论坛体验仍未完全对齐

虽然主流程已可用，但距离 Flarum 仍有体验差距：

- Composer 工具栏、格式化能力、异常提示和边界交互仍不如 Flarum 完整
- 讨论页 Scrubber、`near` 定位、长帖流导航、移动端细节仍有差距
- 版主内联操作、审核动作、举报处理的论坛端体验还不够 Flarum 化
- 帖子恢复、审核前后状态流转、版主批量处理等能力仍偏弱

### 5.2 标签体系仍停留在“能用”，没有达到 Tags 扩展级别

当前标签能力已经有基础，但与 Flarum 官方 `tags` 扩展相比仍差明显一段：

- 标签级权限模型不完整
- 标签范围的发帖/回帖/查看/审核权限没有真正对齐 Flarum
- 父子标签约束、主标签/次标签规则、数量约束仍不够完整
- 标签管理页仍偏 CRUD，不够接近 Flarum 的产品化体验

### 5.3 后台设置与运行时行为尚未完全闭环

这是当前判断完成度时最容易被高估的一块。

后台高级设置中已经能保存很多字段，但并不等于这些能力已经落地。例如：

- `maintenance_mode`
- `maintenance_message`
- `debug_mode`
- `log_queries`
- `queue_enabled`
- `queue_driver`
- `cache_lifetime`

这些字段可见于：

- [AdvancedPage.vue](/D:/files/project/tmp/pyflarum/frontend/src/admin/views/AdvancedPage.vue)
- [settings_service.py](/D:/files/project/tmp/pyflarum/apps/core/settings_service.py#L40)

但从当前代码实现看，它们大量仍属于“设置持久化”而非“系统运行时强制生效”。

这是后续必须优先治理的问题。

### 5.4 官方扩展覆盖仍有明显缺口

本地 Flarum 安装的官方扩展见 [composer.json](/D:/files/project/tmp/flarum/composer.json#L20)，包括：

- `approval`
- `bbcode`
- `emoji`
- `flags`
- `gdpr`
- `likes`
- `lock`
- `markdown`
- `mentions`
- `messages`
- `nicknames`
- `realtime`
- `statistics`
- `sticky`
- `subscriptions`
- `suspend`
- `tags`

`pyflarum` 当前并没有把这些 bundled 能力完整覆盖。

### 5.5 后台产品化深度仍弱于 Flarum

Flarum 官方文档把设置、权限、外观、邮件、扩展、主题、语言都视为一等产品能力：

- <https://docs.flarum.org/2.x/admin/>
- <https://docs.flarum.org/2.x/extensions/>
- <https://docs.flarum.org/2.x/languages/>
- <https://docs.flarum.org/2.x/themes/>
- <https://docs.flarum.org/2.x/mail/>

相比之下，`pyflarum` 目前的后台还存在这些差距：

- 没有 Python 语境下的扩展/插件管理机制
- 没有语言包管理闭环
- 没有主题管理与编译闭环
- 没有更强的统计、趋势、系统健康视图
- 权限模型和标签权限联动不够深入

### 5.6 实时能力仍较基础

目前主要是通知 WebSocket，有价值，但还不等于 Flarum `realtime` 的完整能力。

缺口主要包括：

- 讨论流实时刷新
- 更细粒度的通知实时更新
- 在线状态
- 输入中状态
- 更完整的实时前端状态管理

### 5.7 工程体系仍需加固

虽然已经有 52 个后端测试通过，但和“长期可持续迭代”的工程要求相比仍有短板：

- 没有前端单测
- 没有端到端测试
- 没有前后端 API 契约校验体系
- 没有明确的模块验收标准
- 文档与实际完成度口径不一致

## 8. 建议的后续开发顺序

接下来建议按“先补闭环，再补治理，再预留扩展边界，再补官方扩展，再做正式生态平台”的顺序推进。

### 阶段 A：先补已有能力的断链与假闭环

目标：把“看起来做了，但实际上没完全生效”的部分先做实。

建议优先项：

1. 设置运行时闭环
2. 标签权限与标签约束闭环
3. 论坛端审核/举报/封禁流程打通
4. Composer、Scrubber、移动端细节继续对齐
5. 前后端关键接口增加回归测试

验收标准：

- 设置项保存后能真实影响系统行为
- 标签权限可影响讨论列表、发帖、回帖、审核与后台配置
- 审核/举报/封禁不只后台可见，论坛端也形成完整治理体验

### 阶段 B：补齐 Flarum 官方治理能力

目标：从“能发帖的论坛”升级到“能稳定运营社区的论坛”。

建议优先项：

1. `approval` 深化
2. `flags` 深化
3. `suspend` 深化
4. 更完整的通知偏好和治理反馈

验收标准：

- 版主在论坛前台即可处理大部分治理动作
- 用户能明确感知审核、举报、封禁的状态变化
- 关键治理路径都具备后端测试

### 阶段 C：先做轻量级扩展边界预留

目标：现在就把未来插件化会用到的接缝预留出来，但不在这个阶段做完整插件平台。

建议优先项：

1. 权限注册入口
2. 后台菜单注册入口
3. 前台路由或页面注入入口
4. 设置项命名空间约定
5. 通知类型注册约定
6. 事件、信号或 hooks 的最小机制

验收标准：

- 新能力不再全部写死在单体主流程里
- 后续官方扩展模块开发时可以复用统一接缝
- 不引入过重的插件生命周期和管理复杂度

### 阶段 D：补齐官方扩展中的高价值缺口

建议优先顺序：

1. `nicknames`
2. `statistics`
3. `realtime`
4. `messages`
5. `gdpr`

排序理由：

- `nicknames` 和 `statistics` 能较快提升“像 Flarum”的产品感知
- `realtime` 能显著提升使用体验，但技术复杂度高于前两者
- `messages` 和 `gdpr` 都是独立产品模块，应在核心治理能力稳定后再做

### 阶段 E：后台生态与正式 Python 化扩展体系

这是 `pyflarum` 和 Flarum 真正拉开差距、也最需要“按 Python 语境重新设计”的部分，但不应早于前面的核心闭环、治理能力和扩展边界预留。

建议方向：

1. 在已预留的扩展边界之上，设计正式的 Python 插件/扩展机制
2. 建立语言包管理策略
3. 建立主题/样式扩展机制
4. 建立扩展安装、启停、配置、文档展示的后台入口

## 9. 按阶段排期的完整任务池

第 9 节用于承接第 8 节的阶段顺序，把后续所有待开发和待完善事项都落成明确任务池。后续开发原则上按本节顺序推进；若中途调整优先级，应同步修改本节。

### 下一轮立即开做的 P0 清单

以下任务建议作为下一轮开发的直接入口，按顺序推进，不建议并行铺开过多模块。

#### P0-1：设置运行时闭环

目标：

- 先解决“后台能保存，但运行时不生效”的问题

本轮最低交付：

- [ ] `maintenance_mode` 生效
- [ ] `maintenance_message` 生效
- [ ] `log_queries` 生效或明确废弃
- [ ] `queue_enabled` / `queue_driver` 语义澄清
- [ ] `cache_lifetime` 接入实际缓存逻辑
- [ ] 补齐后端测试

完成判定：

- 管理员设置后能在前台或 API 层观察到真实行为变化
- 文档中明确哪些配置即时生效，哪些配置需要重启

#### P0-2：标签权限与标签约束闭环

目标：

- 让标签不再只是分类，而是实际参与权限和内容组织

本轮最低交付：

- [ ] 标签级查看权限
- [ ] 标签级发帖权限
- [ ] 标签级回帖权限
- [ ] 主标签/次标签数量约束
- [ ] 父子标签组合校验
- [ ] 标签权限回归测试

完成判定：

- 列表、详情、Composer、搜索结果在标签权限上行为一致
- 非法标签组合会在前后端同时被拦截

#### P0-3：论坛端治理链路打通

目标：

- 让审核、举报、封禁从“后台可操作”升级为“用户和版主都能感知的完整链路”

## 10. 后续开发原则

### 10.1 产品体验对齐优先于技术实现对齐

不要为了“更像 Flarum 的内部实现”而牺牲 Python 项目的可维护性。

要对齐的是：

- 用户感知到的论坛行为
- 管理员感知到的后台能力
- 官方扩展对应的产品闭环

不必强行对齐的是：

- PHP 内部架构
- Composer 扩展安装方式
- 原项目具体技术栈约束

### 10.2 前后端必须同步闭环

每个功能都必须同时考虑：

- 数据模型
- 权限与策略
- API 契约
- 前台 UI
- 后台 UI
- 通知/邮件/实时反馈
- 测试

只做其中一两层，不算完成。

### 10.3 先预留扩展边界，再做正式插件平台

插件和扩展不应在最前面就做成完整平台，也不应等所有模块完全固化后再考虑。

推荐策略：

- 前期先预留轻量级扩展接缝
- 中期用 1 到 2 个真实模块验证扩展接缝是否合理
- 后期再建设正式插件平台与扩展管理后台

这样可以避免两类问题：

- 因为过早平台化而偏离核心产品闭环
- 因为完全不预留接缝而导致后期难以插件化

### 10.4 小步迭代，一次只收敛一个模块

后续不建议再做“大而散”的横向铺开。

推荐方式：

- 一次只选一个模块
- 先补最核心闭环
- 做完即补测试
- 做完即更新本文档

### 10.5 优先修复用户可感知的断点

优先级判断标准：

1. 用户是否会直接遇到断链
2. 管理员是否会遇到“设置了但没生效”
3. 论坛是否缺少治理能力
4. 是否显著影响后续扩展开发

### 10.6 把本地 Flarum 与官方文档作为验收基线

每做一个模块，都应同时参考：

- 本地 Flarum 运行效果
- 官方文档说明
- 官方社区中的实际使用语境

不要只凭感觉“猜一个差不多的 Flarum 风格”。

### 10.7 用能力矩阵和验收标准说话

以后不再使用“接近完成”“大致支持”这种模糊口径作为里程碑判断。

应该使用：

- 是否有前台入口
- 是否有后台入口
- 是否真正影响运行时
- 是否有权限控制
- 是否有测试覆盖
- 是否能对标 Flarum 对应产品行为

## 11. Definition of Done

后续每个模块完成时，至少满足以下条件：

1. 有明确的数据模型或状态模型
2. 有明确的权限控制
3. 有稳定 API 契约
4. Forum 端入口完整
5. Admin 端入口完整
6. 设置项若存在，必须真实影响运行时
7. 有必要的通知、邮件或实时反馈
8. 至少覆盖主路径与关键失败路径测试
9. 文档与路线图同步更新

若缺任一关键项，只能算“部分完成”。

## 12. 后续维护要求

后续每完成一轮开发，必须同步更新本文档：

- 更新整体完成度判断
- 更新模块状态
- 删除已完成的差距项
- 为下一轮迭代重新排序优先级

建议把本文档视为项目真实状态，而不是以 README 的宣传性描述作为完成度依据。

## 13. 代码审查与架构审查流程

后续每一轮开发完成后，除功能验收外，还应固定执行一次代码审查。代码审查不只看功能是否可用，还要看安全性、可维护性和架构一致性。

### 13.1 审查顺序

建议按以下顺序执行：

1. 功能行为审查
2. 权限与安全审查
3. 数据一致性审查
4. 接口契约审查
5. 前端状态与交互审查
6. 性能与资源使用审查
7. 整体架构审查
8. 测试与回归审查
9. 文档同步审查

### 13.2 安全漏洞审查清单

- [ ] 认证是否可靠，是否存在未登录可访问敏感接口的问题
- [ ] 权限是否只在前端控制，后端是否也做了强校验
- [ ] 是否存在越权访问、越权编辑、越权删除、越权审核问题
- [ ] 输入内容是否做了必要校验和清洗
- [ ] 富文本、Markdown、附件、头像、搜索输入是否存在 XSS 风险
- [ ] 上传接口是否校验文件类型、大小、扩展名和访问路径
- [ ] 是否存在敏感配置泄露，例如密钥、邮箱口令、对象存储凭证
- [ ] 错误信息是否泄露内部实现细节
- [ ] WebSocket 是否存在未鉴权订阅或错误广播
- [ ] 是否存在明显的接口滥用风险，例如暴力请求、批量刷接口、无分页拉全量数据

### 13.3 数据与业务一致性审查清单

- [ ] 状态流转是否完整，例如审核、举报、封禁、已读、关注、通知
- [ ] 数据写入后是否会造成计数、列表、缓存、通知不一致
- [ ] 删除、恢复、隐藏、封禁等动作是否影响了所有关联对象
- [ ] 是否存在事务缺失导致的半成功状态
- [ ] 是否存在后台已修改但前台仍读旧值的问题

### 13.4 接口与前端闭环审查清单

- [ ] API 返回结构是否稳定
- [ ] 前端是否依赖了脆弱字段或隐式字段
- [ ] 错误码、错误文案、空态是否一致
- [ ] 权限不足、资源不存在、参数非法等失败路径是否有明确处理
- [ ] 新增设置项是否同时有后台入口、运行时行为和回归测试

### 13.5 架构审查清单

- [ ] 新功能是否放在正确的应用边界中，没有把职责继续堆到单一文件
- [ ] 是否出现过度耦合，例如前端页面直接依赖过多后端内部细节
- [ ] 是否复用了已有服务层、权限层、通知层，而不是重复造轮子
- [ ] 是否为未来扩展预留了合理接缝，而不是继续硬编码
- [ ] 模型、服务、API、前端、后台之间的责任划分是否清晰
- [ ] 是否引入了难以维护的特殊分支或临时逻辑
- [ ] 代码命名、目录结构、模块边界是否仍然一致

### 13.6 性能与稳定性审查清单

- [ ] 是否引入明显的 N+1 查询
- [ ] 列表接口是否有分页、筛选和必要索引支持
- [ ] 搜索、通知、统计、实时功能是否有明显可预见的性能瓶颈
- [ ] 文件上传、图片加载、WebSocket 连接是否存在明显资源浪费
- [ ] 缓存策略是否与实际业务一致
- [ ] 是否考虑了失败重试、超时和降级路径

### 13.7 审查产出要求

每次代码审查结束后，至少产出以下内容：

- 本轮发现的问题清单
- 问题严重级别排序
- 是否阻塞合并
- 需要补充的测试
- 需要补充的文档
- 是否影响后续架构方向

若发现高风险安全问题、越权问题、数据一致性问题或架构性回退，应优先修复，不应带着问题继续叠加新功能。
