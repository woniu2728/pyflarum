这份文档用于统一 `pyflarum` 后续开发的评估口径、差距清单、迭代顺序和开发原则，避免继续以“页面已经有了”或“字段已经存了”来误判完成度。

本次不是对标 Flarum 的 PHP/Composer 实现细节，而是对标 Flarum 2.x 的“官方产品体验”与“官方扩展能力闭环”。

- 产品体验对齐优先于技术实现对齐。
- 前端、后端、权限、通知、设置运行时、测试，必须形成闭环，才算完成。
- “有页面但不能用”、“有设置但不生效”、“有接口但没有前端入口”，都只能算部分完成。

## 3. 总体结论

### 3.1 当前项目定位

`pyflarum` 当前已经不是“论坛骨架”，而是一个具备较强可用性的 Flarum 风格论坛系统；但它距离“Flarum 2.x 官方产品体验的 Python 复刻版”仍有明确差距。

它更接近：

- 一个已有较完整核心论坛链路的 Django + Vue 社区产品
- 一个部分复刻了 Flarum 前台体验与后台结构的项目

它还不接近：

- README 所写的“100% 完成”或“完美复刻”
- Flarum 2.x 官方内置产品体验的完整等价实现

README 当前口径见：[README.md](/D:/files/project/tmp/pyflarum/README.md#L1)

### 3.2 完成度估算

以下是按“Flarum 2.x 官方产品体验”口径给出的保守估算：

| 维度 | 当前估算 | 判断 |
| --- | --- | --- |
| 前台核心论坛体验 | 75% - 80% | 讨论、帖子、基础通知、搜索、Composer 已较完整，但细节与治理能力仍弱 |
| 后台产品化能力 | 55% - 65% | 后台页面齐全度不错，但不少模块仍停留在基础 CRUD 或设置持久化 |
| 官方扩展能力覆盖 | 45% - 55% | `likes`/`lock`/`sticky`/`subscriptions` 较接近，多个官方扩展仍明显缺失 |
| 工程成熟度 | 60% - 70% | 后端测试已有基础，但缺少前端测试、E2E、契约测试和更强运行时验证 |
| 整体对标 Flarum 2.x | 65% - 70% | 已有明显成果，但还不能宣称“完整复刻” |

结论：

- 可以把当前项目视为“已完成约三分之二的 Flarum 2.x Python 化产品复刻”
- 下一阶段重点应收敛为“补治理、补安装初始化、补文档、补后台体验，并继续补运行时与测试”

## 5. PyFlarum 与 Flarum 2.x 的主要差距

以下差距是当前最需要正视的部分。

### 5.1 核心论坛体验仍未完全对齐

虽然主流程已可用，但距离 Flarum 仍有体验差距：

- Composer 工具栏、格式化能力、异常提示和边界交互仍不如 Flarum 完整
- 讨论页 Scrubber、`near` 定位、长帖流导航、移动端细节仍有差距
- 版主内联操作、审核动作、举报处理的论坛端体验还不够 Flarum 化
- 帖子恢复、审核前后状态流转、版主批量处理等能力仍偏弱

### 5.2 标签体系仍停留在“能用”，没有达到 Tags 扩展级别

当前标签能力已经有基础，但与 Flarum 官方 `tags` 扩展相比仍差明显一段：

- 标签级权限模型不完整
- 标签范围的发帖/回帖/查看/审核权限没有真正对齐 Flarum
- 父子标签约束、主标签/次标签规则、数量约束仍不够完整
- 标签管理页仍偏 CRUD，不够接近 Flarum 的产品化体验

### 5.3 后台设置与运行时行为尚未完全闭环

这是当前判断完成度时最容易被高估的一块。

后台高级设置中已经能保存很多字段，但并不等于这些能力已经落地。例如：

- `maintenance_mode`
- `maintenance_message`
- `debug_mode`
- `log_queries`
- `queue_enabled`
- `queue_driver`
- `cache_lifetime`

这些字段可见于：

- [AdvancedPage.vue](/D:/files/project/tmp/pyflarum/frontend/src/admin/views/AdvancedPage.vue)
- [settings_service.py](/D:/files/project/tmp/pyflarum/apps/core/settings_service.py#L40)

但从当前代码实现看，它们大量仍属于“设置持久化”而非“系统运行时强制生效”。

这是后续必须优先治理的问题。

### 5.4 官方扩展覆盖仍有明显缺口

本地 Flarum 安装的官方扩展见 [composer.json](/D:/files/project/tmp/flarum/composer.json#L20)，包括：

- `approval`
- `bbcode`
- `emoji`
- `flags`
- `gdpr`
- `likes`
- `lock`
- `markdown`
- `mentions`
- `messages`
- `nicknames`
- `realtime`
- `statistics`
- `sticky`
- `subscriptions`
- `suspend`
- `tags`

`pyflarum` 当前并没有把这些 bundled 能力完整覆盖。

### 5.5 后台产品化深度仍弱于 Flarum

Flarum 官方文档把设置、权限、外观、邮件、扩展、主题、语言都视为一等产品能力：

- <https://docs.flarum.org/2.x/admin/>
- <https://docs.flarum.org/2.x/extensions/>
- <https://docs.flarum.org/2.x/languages/>
- <https://docs.flarum.org/2.x/themes/>
- <https://docs.flarum.org/2.x/mail/>

相比之下，`pyflarum` 目前的后台还存在这些差距：

- 没有 Python 语境下的扩展/插件管理机制
- 没有语言包管理闭环
- 没有主题管理与编译闭环
- 没有更强的统计、趋势、系统健康视图
- 权限模型和标签权限联动不够深入

### 5.6 实时能力仍较基础

目前主要是通知 WebSocket，有价值，但还不等于 Flarum `realtime` 的完整能力。

缺口主要包括：

- 讨论流实时刷新
- 更细粒度的通知实时更新
- 在线状态
- 输入中状态
- 更完整的实时前端状态管理

### 5.7 工程体系仍需加固

虽然已经有 94 个后端测试通过，但和“长期可持续迭代”的工程要求相比仍有短板：

- 没有前端单测
- 没有端到端测试
- 没有前后端 API 契约校验体系
- 没有明确的模块验收标准
- 文档与实际完成度口径不一致

## 8. 当前版本后的主线调整

自 2026-04-17 起，项目主线不再沿用之前“先扩展接缝、再官方扩展、再生态平台”的推进方式。

当前版本后的唯一主线，调整为以下四个方向：

1. 完善治理链路
2. 优化安装、初始化与升级方式
3. 重写 README，按完整项目口径提供介绍与安装说明
4. 优化后台体验

这意味着以下方向从“当前主线”调整为“后置事项 / 冻结事项”：

- `messages`
- `gdpr`
- `nicknames`
- `statistics`
- `realtime`
- 正式插件 / 扩展机制
- 语言包管理
- 主题生态平台化

说明：

- 这些能力不是永久放弃，而是在当前阶段不再占用主线开发带宽
- 当前阶段优先级判断不再以“补齐 bundled 官方扩展”作为主驱动，而以“把论坛作为完整项目做扎实”作为主驱动
- 只有当治理、安装、文档、后台体验四条主线达到可接受状态后，才重新评估这些后置能力

### 8.1 当前主线一：完善治理链路

目标：

- 把审核、举报、封禁从“已具备基础能力”推进到“站长、版主、普通用户都能顺畅理解和使用”的完整链路

当前判断：

- 这是当前最有必要继续投入的方向
- 相比新扩展开发，它更直接影响论坛能否稳定运营

### 8.2 当前主线二：优化安装、初始化与升级方式

目标：

- 让新用户或部署者可以更容易完成首次安装、数据库选择、管理员创建、基础配置以及后续版本升级

当前判断：

- 这是把项目从“开发仓库”提升为“可分发项目”的关键一步
- 安装便捷性当前明显弱于项目已有功能成熟度
- 如果缺少稳定升级能力，项目即使能安装，也很难作为长期运营的软件交付

### 8.3 当前主线三：重写 README

目标：

- 去掉当前 README 中不准确的“100% 完成 / 完美复刻”口径，按一个真实可安装、可部署、可维护的项目来写

当前判断：

- 这是项目对外表达层的优先项，不再是可选文档工作
- README 必须和当前真实能力、安装方式、依赖要求保持一致

### 8.4 当前主线四：优化后台体验

目标：

- 把后台从“页面齐全、功能基本可用”进一步推进到“真正适合长期运营和日常配置”

当前判断：

- 这是当前最能直接提升产品观感和可运营性的方向之一
- 重点不是泛泛补页面，而是把已有后台页做实、做顺、做完整

## 9. 当前主线任务池

本节从 2026-04-17 开始，替代旧的阶段 A-E 推进顺序，作为新的唯一主线任务池。后续原则上只按本节推进；若再次调整优先级，应直接修改本节，而不是在旧任务池上追加补丁。

### 下一轮直接入口

#### P0-1：治理链路收尾与做实

目标：

- 继续把审核、举报、封禁做成可运营的完整链路

本轮最低交付：

- [x] 前台审核状态提示
- [x] 前台版主处理举报入口
- [x] 治理动作基础反馈
- [x] 封禁提示与恢复逻辑补齐
- [x] 审核通过 / 拒绝后的通知与状态反馈补齐
- [x] 权限拒绝路径专项测试
- [x] 举报 / 审核 / 封禁三条链路的状态一致性回归

当前进展补充：

- 讨论页已支持前台审核状态展示、被拒原因展示、前台通过 / 拒绝处理
- 举报原因、举报提交、举报人反馈、版主前台处理举报入口已基本打通
- 作者现在可以看到自己被拒绝的讨论 / 回复及对应审核备注
- 审核通过 / 拒绝现在会向讨论作者或回复作者发送站内通知，并在通知中心提供对应跳转
- 后台封禁 / 解封用户后，现在会向目标用户发送站内通知；前台收到通知后会刷新当前账号状态，减少封禁状态滞后
- 被拒讨论 / 回复现在支持作者在前台直接修改并重新提交审核，不再停留在“只能看到拒绝原因”的半链路状态
- 当前已补齐治理链路关键回归：覆盖审核队列越权拒绝、被拒内容整改重提、审核通过后对其他成员的可见性恢复、整改后列表与详情可见性收敛

完成判定：

- 普通用户、内容作者、版主三种角色都能感知治理状态

## 10. 后续开发原则

### 10.1 产品体验对齐优先于技术实现对齐

不要为了“更像 Flarum 的内部实现”而牺牲 Python 项目的可维护性。

要对齐的是：

- 用户感知到的论坛行为
- 管理员感知到的后台能力
- 官方扩展对应的产品闭环

不必强行对齐的是：

- PHP 内部架构
- Composer 扩展安装方式
- 原项目具体技术栈约束

### 10.2 前后端必须同步闭环

每个功能都必须同时考虑：

- 数据模型
- 权限与策略
- API 契约
- 前台 UI
- 后台 UI
- 通知/邮件/实时反馈
- 测试

只做其中一两层，不算完成。

### 10.3 当前阶段不推进扩展平台化

插件和扩展不是当前阶段主线，不应继续占用治理、安装、文档和后台优化的开发带宽。

当前策略：

- 当前阶段先不设计正式插件平台
- 当前阶段先不新增扩展接缝任务
- 待四条主线达到可接受状态后，再重新评估是否需要扩展机制、语言包管理和主题生态能力

这样可以避免两类问题：

- 因为过早平台化而偏离当前最关键的产品闭环
- 因为路线摇摆导致文档和实际开发再次脱节

### 10.4 小步迭代，一次只收敛一个模块

后续不建议再做“大而散”的横向铺开。

推荐方式：

- 一次只选一个模块
- 先补最核心闭环
- 做完即补测试
- 做完即更新本文档

### 10.5 优先修复用户可感知的断点

优先级判断标准：

1. 用户是否会直接遇到断链
2. 管理员是否会遇到“设置了但没生效”
3. 论坛是否缺少治理能力
4. 是否显著影响后续治理、安装或后台优化主线

### 10.6 把本地 Flarum 与官方文档作为验收基线

每做一个模块，都应同时参考：

- 本地 Flarum 运行效果
- 官方文档说明
- 官方社区中的实际使用语境

不要只凭感觉“猜一个差不多的 Flarum 风格”。

### 10.7 用能力矩阵和验收标准说话

以后不再使用“接近完成”“大致支持”这种模糊口径作为里程碑判断。

应该使用：

- 是否有前台入口
- 是否有后台入口
- 是否真正影响运行时
- 是否有权限控制
- 是否有测试覆盖
- 是否能对标 Flarum 对应产品行为

## 11. Definition of Done

后续每个模块完成时，至少满足以下条件：

1. 有明确的数据模型或状态模型
2. 有明确的权限控制
3. 有稳定 API 契约
4. Forum 端入口完整
5. Admin 端入口完整
6. 设置项若存在，必须真实影响运行时
7. 有必要的通知、邮件或实时反馈
8. 至少覆盖主路径与关键失败路径测试
9. 文档与路线图同步更新

若缺任一关键项，只能算“部分完成”。

## 13. 代码审查与架构审查流程

后续每一轮开发完成后，除功能验收外，还应固定执行一次代码审查。代码审查不只看功能是否可用，还要看安全性、可维护性和架构一致性。

### 13.1 审查顺序

建议按以下顺序执行：

1. 功能行为审查
2. 权限与安全审查
3. 数据一致性审查
4. 接口契约审查
5. 前端状态与交互审查
6. 性能与资源使用审查
7. 整体架构审查
8. 测试与回归审查
9. 文档同步审查

### 13.2 安全漏洞审查清单

- [ ] 认证是否可靠，是否存在未登录可访问敏感接口的问题
- [ ] 权限是否只在前端控制，后端是否也做了强校验
- [ ] 是否存在越权访问、越权编辑、越权删除、越权审核问题
- [ ] 输入内容是否做了必要校验和清洗
- [ ] 富文本、Markdown、附件、头像、搜索输入是否存在 XSS 风险
- [ ] 上传接口是否校验文件类型、大小、扩展名和访问路径
- [ ] 是否存在敏感配置泄露，例如密钥、邮箱口令、对象存储凭证
- [ ] 错误信息是否泄露内部实现细节
- [ ] WebSocket 是否存在未鉴权订阅或错误广播
- [ ] 是否存在明显的接口滥用风险，例如暴力请求、批量刷接口、无分页拉全量数据

### 13.3 数据与业务一致性审查清单

- [ ] 状态流转是否完整，例如审核、举报、封禁、已读、关注、通知
- [ ] 数据写入后是否会造成计数、列表、缓存、通知不一致
- [ ] 删除、恢复、隐藏、封禁等动作是否影响了所有关联对象
- [ ] 是否存在事务缺失导致的半成功状态
- [ ] 是否存在后台已修改但前台仍读旧值的问题

### 13.4 接口与前端闭环审查清单

- [ ] API 返回结构是否稳定
- [ ] 前端是否依赖了脆弱字段或隐式字段
- [ ] 错误码、错误文案、空态是否一致
- [ ] 权限不足、资源不存在、参数非法等失败路径是否有明确处理
- [ ] 新增设置项是否同时有后台入口、运行时行为和回归测试

### 13.5 架构审查清单

- [ ] 新功能是否放在正确的应用边界中，没有把职责继续堆到单一文件
- [ ] 是否出现过度耦合，例如前端页面直接依赖过多后端内部细节
- [ ] 是否复用了已有服务层、权限层、通知层，而不是重复造轮子
- [ ] 是否为未来扩展预留了合理接缝，而不是继续硬编码
- [ ] 模型、服务、API、前端、后台之间的责任划分是否清晰
- [ ] 是否引入了难以维护的特殊分支或临时逻辑
- [ ] 代码命名、目录结构、模块边界是否仍然一致

### 13.6 性能与稳定性审查清单

- [ ] 是否引入明显的 N+1 查询
- [ ] 列表接口是否有分页、筛选和必要索引支持
- [ ] 搜索、通知、统计、实时功能是否有明显可预见的性能瓶颈
- [ ] 文件上传、图片加载、WebSocket 连接是否存在明显资源浪费
- [ ] 缓存策略是否与实际业务一致
- [ ] 是否考虑了失败重试、超时和降级路径

### 13.7 审查产出要求

每次代码审查结束后，至少产出以下内容：

- 本轮发现的问题清单
- 问题严重级别排序
- 是否阻塞合并
- 需要补充的测试
- 需要补充的文档
- 是否影响后续架构方向

若发现高风险安全问题、越权问题、数据一致性问题或架构性回退，应优先修复，不应带着问题继续叠加新功能。
